# ⚡ 《Grid Master: Cyber Network》v1.1 需求規格（遊玩回饋修訂版）

> 來源：2026-07-12 正式站上線後首輪實測回饋（六條）。本文件只描述 v1.0 → v1.1 的增量需求；
> 基礎規格仍以 [PRD.md](PRD.md) 為準，實作時一併修訂其中被本文標註的條文。

---

## 1. 需求總覽與優先序

| 編號 | 需求 | 層面 | 建議里程碑 |
|------|------|------|-----------|
| R2 | 自有／他人電廠可查看完整卡牌資訊 | 前端 UI | M8 |
| R3 | 資源市場存量與價格梯常駐面板 | 前端 UI | M8 |
| R4 | 供電收入表可檢視 | 前端 UI | M8 |
| R5 | 規則說明彈窗 | 前端 UI | M8 |
| R1 | 地圖可縮放平移、節點加大 | 前端 PixiJS | M9 |
| R6 | 帳密註冊／登入（測試用） | 後端＋DB＋部署 | M10 |

排序理由：R2–R5 全是「資訊透明化」的純前端呈現，資料來源（`/api/static` 的
rules/deck 與 `room_sync` 的 `resource_market`）**後端都已經在送了**，改動風險最低、
對可玩性提升最大，先做。R1 是獨立的渲染層改動，次之。R6 動到後端、首次真正啟用
Postgres、還牽動部署（migration），單獨收尾。

## 2. 需求細節

### R1 地圖體驗：縮放、平移、節點加大

**現況**（`frontend/src/game/board.js`）：42 城 87 邊等比壓進容器單屏，節點半徑
13px、城市名 10px、過路費 9px，東岸密集區節點重疊、連線與費用難以辨識。

**需求**：
- 地圖世界座標放大為固定虛擬尺寸（例如 1600×1000），節點半徑加大到 ~20px、
  字級同步放大；密集區在世界座標下自然拉開。
- 支援 **滑鼠拖曳平移、滾輪縮放（約 0.5×–3×）、觸控拖曳與 pinch 縮放**。
- 初始載入自動 fit 全圖；提供「回到全圖」按鈕。
- 縮放平移後，點擊節點的命中判定與 hover tooltip 座標必須正確。

**實作備註**：建議手刻 pan/zoom（root `Container` 的 position/scale ＋ pointer
事件，約百行），不引入 pixi-viewport 依賴。注意現有「全量重繪」策略：viewport
transform 要掛在重繪不會重置的外層容器上。

**驗收**：桌機與手機上都能平移縮放；放大後東岸每條邊的過路費可讀；縮放狀態下
擴建點擊落在正確城市。

### R2 電廠詳情查看

**現況**（`frontend/src/components/PlayerStrip.vue`）：玩家列只顯示 `#13🍃` 迷你
標籤，看不到燃料需求（requires）與供電數（powers）；只有官僚階段的 ActionDock
會出現自己電廠的完整卡。

**需求**：
- 點擊任何玩家列上的迷你電廠標籤 → 彈出該卡完整 `PlantCard`（含燃料類型、
  需求數、供電節點數）。自己與他人的電廠都可查。
- 自己的電廠在**任何階段**都能一眼看到完整資訊（常駐「我的設施」區塊或
  hover 即顯示，擇一）。

**驗收**：任何階段兩步以內能查到自己每座電廠的 requires/powers；他人電廠同樣可查。

### R3 資源市場常駐面板

**現況**：`room_sync` 已送 `resource_market[resource].count`，但 UI 上完全沒有
呈現——只有輪到自己採購時，stepper 旁的動態價格能間接推知存量。

**需求**：
- 新增常駐 `ResourceMarket` 面板（所有人含旁觀者可見），畫出**價格梯格位圖**：
  - 💧🔥♻️：$1–$8 每價位 3 格；
  - 🧠：$1–$8 每價位 1 格＋$10/$12/$14/$16 各 1 格。
- 以「填滿／空格」呈現目前存量（價梯不變量：存量永遠填在最貴端連續格位，
  `frontend/src/game/ladder.js` 已有推導邏輯可直接複用）。
- 標示每種資源「下一顆單價」。

**驗收**：任何時刻、任何身份都能看到四種資源的存量分佈與下一顆價格。

### R4 供電收入表

**現況**：`rules.payout` 已在前端（`stores/staticData.js` 的 `payout` getter），
僅用於官僚階段計算單一數字，玩家看不到完整對照表。

**需求**：
- 呈現「供電 0–20 節點 → 收入 $X」完整對照表（放進 R5 規則彈窗，並在官僚
  階段的供電面板旁提供快速入口）。
- 官僚階段中，高亮自己當前選擇可達的那一列。

**驗收**：玩家在選擇啟動設施時能對照收入表看到「多供 1 城多賺多少」。

### R5 規則說明彈窗

**需求**：
- 新增 `RulesModal.vue`，遊戲中任何時刻可開啟（header 常駐「📖 規則」鈕），
  不干擾盤面。分頁建議：
  1. **流程總覽**——回合五階段（順位→競標→採購→擴建→官僚）與反序規則；
  2. **競標**——提名、加價、棄權、電廠上限與棄置；
  3. **資源採購**——價格梯機制＋**補貨對照表**（`rules.resupply` 按人數×Step，
     動態產表）；
  4. **擴建**——進場費（$10/$15/$20 與 Step 開格）、過路費最短路徑、Step 2/3
     觸發條件；
  5. **供電與收入**——R4 收入表、燃料消耗規則、綠能自持；
  6. **遊戲結束**——終局觸發與勝利判定；
  7. **關於**——換皮對照表＋ PRD §7 非商業聲明（與 footer 同文）。
- 表格類內容（補貨表、收入表、卡牌一覽）從 `/api/static` 動態產生，避免與
  後端數據二次維護；敘述文字為手寫靜態內容。

**驗收**：不看外部資料、只靠彈窗能理解一整回合怎麼進行；表格數值與後端
`game_rules.json` 一致（動態產生保證）。

### R6 帳密註冊／登入（測試用輕量帳號）

**動機**：多帳號測試目前得開多個 Discord 帳號＋無痕視窗，太麻煩。

**定位**：輕量帳號系統——**無 email、無驗證信、無密碼找回**；UI 上明示「測試
用帳號，請勿使用你在其他服務的密碼」。

**規則定案（2026-07-12）**：帳號**純英數 3–20 字**（不用 email、禁符號）；
密碼**至少 4 字**；rate limit **先不做**。

**後端**（Postgres 首次真正啟用——當初保留 Ecto 的用途）：
- `accounts` 資料表：`id`、`username`（唯一、不分大小寫）、`password_hash`、
  `inserted_at`。密碼以 **argon2**（`argon2_elixir`）雜湊，明文不落地不留 log。
- `POST /api/auth/register` `{username, password}`：依上述定案驗證；
  成功即視同登入。
- `POST /api/auth/login`：驗證後回傳 **Phoenix.Token**（新 salt
  `"password_auth"`、30 天效期，與 Discord token 同模式）＋顯示名。
- `UserSocket` 接受此 token → 身份 `p_<account_id>`、role `user`，與 Discord
  使用者同權（可入座）；壞 token 退回訪客，沿用現行行為。
- 保留現有 `alias_of` 訪客合併機制（訪客旁觀中→登入→身份無縫接手）。
- ~~（可選）註冊 rate limit~~（2026-07-12 定案：先不做）。

**PRD 條文修訂**：
- §3.2 入座門檻：「僅 Discord User 與 Admin 可入座」→「僅**已登入使用者**
  （Discord 或註冊帳號）與 Admin 可入座；Guest 只能旁觀與聊天」。
- §3.3 斷線重連：「透過 Discord ID 綁定」→「透過登入身份 ID 綁定」。

**部署影響**：
- 新增 `GridMaster.Release.migrate/0`，容器啟動時自動跑 migration
  （`bin/grid_master eval` 或 entrypoint），CI 測試環境已有 Postgres service
  不需改。
- 正式站 db sidecar 與 `DATABASE_URL` 已就緒，首次部署 v1.1 時需確認
  `grid_master_prod` 資料庫已建立（migration 前置）。

**驗收**：兩個無痕視窗各註冊一帳號即可入座對戰；錯誤密碼登入失敗且訊息不透露
帳號是否存在；偽造／過期 token 退回訪客；Discord 身份與帳密身份互不干擾。

## 3. 里程碑切分與停點

| 里程碑 | 內容 | 停點（需使用者確認） |
|--------|------|---------------------|
| **M8 資訊透明化** | R2 電廠詳情＋R3 資源市場面板＋R4 收入表＋R5 規則彈窗 | 完成後實際遊玩驗收 UI 佈局與規則文案 |
| **M9 地圖體驗** | R1 縮放平移＋節點加大 | 完成後桌機＋手機實測手感 |
| **M10 帳密登入** | R6 全套（DB→API→Socket→前端表單→migration 部署） | 動工前確認：密碼政策、rate limit 要不要、§3.2 修訂文字 |

M8、M9 純前端，部署零風險（image 換版即可）；M10 是首次帶 DB schema 的部署，
排最後、單獨上線觀察。

## 4. 非目標（v1.1 明確不做）

- 台灣地圖、動態地圖編輯器（維持 PRD 既定的後續版本目標）。
- 正式美術與 Rive 動畫（維持佔位美術）。
- 帳號系統的 email 驗證、密碼找回、個人資料頁。
- 遊戲狀態持久化（斷線重連仍靠 GenServer 記憶體＋身份綁定）。
